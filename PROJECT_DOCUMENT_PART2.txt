3.1 HIGH-LEVEL ARCHITECTURE OVERVIEW
--------------------------------------

The Smart Attendance System is built upon a modern, three-tier web architecture
that cleanly separates the presentation layer (frontend), the business logic and
service layer (backend services and API), and the data persistence layer
(cloud database). This separation ensures that each tier can be scaled, tested,
and modified independently without disrupting the others.

At the highest level, the architecture can be described as follows:

TIER 1 - PRESENTATION LAYER (React Frontend):
The user-facing layer is a Single Page Application (SPA) built with React 18
and TypeScript. The application runs entirely in the browser, providing multiple
distinct views and portals based on the authenticated user's role. The browser
accesses the device camera through the WebRTC MediaDevices API, runs the face
recognition models using TensorFlow.js (WASM backend), and communicates with
backend services through HTTP REST API calls.

TIER 2 - SERVICE LAYER:
Two distinct backend services handle non-browser-side logic:
  (a) Supabase Services: The Supabase platform provides a fully managed
      PostgreSQL database, automatic REST API generation from the schema,
      real-time data subscriptions, and built-in authentication. The frontend
      communicates directly with Supabase through the official JavaScript SDK.
  (b) Email Server: A lightweight Express.js HTTP server running on port 3001
      handles email dispatch via Nodemailer and Gmail SMTP. The React frontend
      sends HTTP POST requests to this server for every email notification event.

TIER 3 - DATA LAYER (Supabase/PostgreSQL):
All persistent data — student profiles, biometric face descriptors, attendance
records, and voice audio data — resides in a hosted PostgreSQL database managed
by Supabase. The database is accessed exclusively through Supabase's PostgREST
interface, which translates the REST API calls from the frontend into SQL queries
and enforces Row Level Security (RLS) policies.

HIGH-LEVEL COMPONENT INTERACTION:

  [Student/Admin Browser]
         |
         |--- WebRTC Camera API (local, no network)
         |--- TensorFlow.js WASM (local, no network)
         |--- Supabase JS SDK ---> [Supabase PostgREST] ---> [PostgreSQL DB]
         |--- fetch() ---------> [Express Email Server] ---> [Gmail SMTP]
                                                                    |
                                                              [Student Email]

3.3 BACKEND ARCHITECTURE

The backend architecture consists of two components:

Supabase is used as a Backend-as-a-Service (BaaS) platform. It provides:

    - Nodemailer createTransport() configured with Gmail SMTP credentials
    - Single endpoint: POST /send-email
    - Four HTML email templates (registered, checkin, checkout, absent)

3.4 DATABASE ARCHITECTURE
--------------------------

The database layer uses PostgreSQL 15 hosted on Supabase's managed cloud
infrastructure. The schema is organized around two primary domain areas:
student identity and attendance events.

PRIMARY TABLES:
  1. students -- Combined user/identity table for all roles
  2. attendance -- Attendance event records with check-in/out data

SCHEMA DESIGN PHILOSOPHY:
The schema deliberately uses a single "students" table to store users of all
roles (student, admin, staff). This simplification is intentional for the initial
system version and avoids complex join structures for a deployment scenario where
admin accounts are few and manually managed. The "role" column distinguishes
between user types for access control logic in the application layer.

3.5 FACE RECOGNITION PIPELINE
-------------------------------

The face recognition pipeline is the core technical innovation of this system.
It operates in two phases: enrollment (during registration) and identification
(during attendance scanning).

ENROLLMENT PHASE (Registration):
Step 1: Camera Capture
  - The browser requests camera access via navigator.mediaDevices.getUserMedia()
  - Multiple frames (minimum 3) are captured as HTML5 Canvas snapshots
  - Each capture is displayed to the user for visual confirmation

Step 2: Face Detection
  - Each captured frame is processed by detectSingleFace() from face-api.js
  - The SSD MobileNet V1 model detects the face bounding box
  - The 68-point facial landmark model identifies specific facial geometry points
    including eye corners, nose tip, mouth corners, and jaw outline

Step 3: Face Descriptor Extraction
  - The FaceRecognitionNet model (ResNet-34 inspired architecture) processes
    the aligned face crop and outputs a 128-dimensional Float32Array
  - This "face descriptor" encodes the unique geometric relationships between
    facial landmarks into a numerical representation
  - Multiple descriptors from multiple captures are averaged for robustness

Step 4: Storage
  - The descriptor array is converted to JSON string via JSON.stringify(Array.from(descriptor))
  - Stored in the face_descriptor column of the students table
  - The student photo URL is also stored for visual identification in the directory

IDENTIFICATION PHASE (Attendance Scanning):
Step 1: Continuous Video Scanning
  - The camera video stream is processed every 400ms via setInterval()
  - detectAllFaces() (using TinyFaceDetector for speed) finds all faces in frame

Step 2: Descriptor Extraction
  - For each detected face, the same 128-dimensional descriptor is extracted
  - The tiny face detector is used here for speed (vs. SSD MobileNet for enrollment)

Step 3: Database Comparison
  - Pre-loaded student descriptors (loaded once on component mount) are stored
    in a React useRef for O(1) access without triggering re-renders
  - For each detected face, Euclidean distance is calculated against every
    stored student descriptor using the compareFaces() function
  - The student with the minimum distance is identified as the best match

Step 4: Threshold Decision
  - If minimum distance <= 0.50: MATCH IDENTIFIED (confidence > 50%)
  - If minimum distance > 0.50: UNKNOWN FACE

Step 5: Attendance Recording
  - If match found AND no existing attendance today: Record check-in, send email
  - If match found AND already checked in but not checked out: Initiate checkout flow
  - If match found AND both check-in and check-out exist: Display "Already Taken"

FACE DESCRIPTOR MATHEMATICS:
The Euclidean distance between two 128-dimensional vectors d1 and d2 is:

  distance = sqrt( sum( (d1[i] - d2[i])^2 ) for i in 0..127 )

A distance of 0 indicates identical descriptors (same person, ideal conditions).
A distance of 0.50 is the empirically determined threshold below which two
descriptors reliably represent the same person. Values above 0.60 reliably
indicate different people.

4.1 FRONTEND TECHNOLOGIES
--------------------------

REACT 18.3 (with TypeScript):
React is a declarative, component-based JavaScript library for building user
interfaces, developed and maintained by Meta (Facebook). Version 18 introduces
automatic batching, concurrent rendering features, and the useTransition hook for
non-blocking state updates. TypeScript integration provides static type checking,
improved IDE autocompletion, and compile-time error detection that dramatically
reduces runtime bugs in complex component interactions.

WHY REACT: The component model naturally mirrors the system's modular structure.
Each attendance portal, dashboard view, and form is a self-contained component
that can be developed, tested, and reasoned about independently. React's virtual
DOM ensures UI updates from real-time attendance events are efficient and do not
cause full page reloads.

VITE 5.4:
Vite is a next-generation build tool that provides near-instant development
server startup using native ES modules (no bundling during development) and
Rollup-based optimized production builds. For a project that iteratively develops
real-time camera components, Vite's hot module replacement (HMR) is essential —
changes to component code are reflected in the browser without losing camera
state or reloading face recognition models.

TAILWINDCSS 3.4:
Tailwind is a utility-first CSS framework that allows styling through composable
utility classes directly in JSX markup. This approach eliminates the overhead of
maintaining separate CSS files for complex, state-driven UI components. The premium
glassmorphism design of the student and admin portals — with backdrop-blur,
gradient backgrounds, and animated scan lines — is implemented efficiently through
Tailwind's responsive utility classes.

LUCIDE-REACT 0.344:
A comprehensive icon library providing over 1,000 consistent, accessible SVG
icons as React components. Used throughout the UI for visual communication:
ShieldCheck for verified attendance, VideoOff for camera controls, UserPlus for
registration, Mail for email notifications, and dozens more.

4.3 AI AND FACE RECOGNITION LIBRARIES
---------------------------------------

@VLADMANDIC/FACE-API 1.7.15:
This library is a heavily optimized fork of the original face-api.js by Vincent
Mühler, maintained by Vladimir Mandic. It implements the following deep learning
models optimized for browser execution:

  (a) SSD MobileNet V1: Single Shot MultiBox Detector adapted for face detection.
      A convolutional neural network that produces bounding boxes and confidence
      scores for all detected faces in a single forward pass. Used during student
      enrollment for high-accuracy face detection.

  (b) Tiny Face Detector: A lightweight, YOLO-inspired convolutional architecture
      optimized for real-time face detection at the cost of slightly reduced
      accuracy. This architecture directly reflects the YOLO paradigm of treating
      detection as a single regression problem across a spatial grid. Used during
      the attendance scanning loop for real-time performance at 400ms intervals.

  (c) Face Landmark Detection (68 points): A CNN that takes the cropped face
      region and outputs 68 (x,y) coordinate pairs corresponding to specific
      anatomical facial landmarks (eye corners, eyebrow outlines, nose bridge,
      nostril positions, lip borders, jaw outline).

  (d) Face Recognition Net (ResNet-34 based): Takes the normalized, aligned
      face patch (aligned using the 68 landmark positions) and produces a
      128-dimensional floating-point descriptor vector. This architecture is
      inspired by Google's FaceNet and produces descriptor embeddings where
      faces of the same person cluster together in the 128-dimensional space.

@TENSORFLOW/TFJS 4.22:
TensorFlow.js is Google's JavaScript ML framework that allows trained neural
network models to run in-browser. The WASM (WebAssembly) backend is used in
this system to achieve near-native computational performance for the matrix
operations underlying face detection and recognition.

WHY BROWSER-SIDE AI:
Processing face recognition in the browser rather than on a server provides:
  - ZERO LATENCY: No round-trip to a remote server for each video frame
  - PRIVACY: Raw video frames and face descriptors never leave the user's device
  - OFFLINE CAPABLE: Recognition works even if internet connection is slow
  - COST: No GPU server rental fees

5.1 UI/UX DESIGN PRINCIPLES
-----------------------------

The user interface of the Smart Attendance System is designed according to modern
web design principles with a strong emphasis on premium aesthetics, accessibility,
and operational clarity. The target users range from first-year students unfamiliar
with technology to administrative staff managing daily operations.

GLASSMORPHISM DESIGN LANGUAGE:
The primary aesthetic is glassmorphism — a design trend characterized by:
  - Semi-transparent panels with backdrop-filter: blur() applied
  - Subtle border highlights on the lighter side of surfaces
  - Deep, rich background gradients (navy to slate to indigo)
  - Floating card elements with significant drop shadows
  - Text with strong contrast against blurred backgrounds

This aesthetic communicates a sense of technological sophistication appropriate
for a biometric security system and creates immediate positive first impressions.

COLOR PALETTE:
  Primary: #2563eb (Blue-600) — Trust, technology, security
  Success: #059669 (Emerald-600) — Verification confirmed, present
  Error: #dc2626 (Red-600) — Verification failed, unauthorized
  Warning: #d97706 (Amber-600) — Absence alerts, warnings
  Background: Gradient from #0f172a to #1e3a8a — Deep professional dark
  Text Primary: #f8fafc — Near-white for dark backgrounds
  Text Secondary: rgba(255,255,255,0.6) — Subdued secondary information

TYPOGRAPHY:
  The Inter font family (via browser system font stack with 'Segoe UI' fallback)
  is used exclusively. Inter was designed by Rasmus Andersson specifically for
  screen readability with excellent legibility at all sizes from 8px to 72px.
  Font weights range from regular (400) for body text to black (900) for
  status notifications and headings.

MOTION AND ANIMATION:
  CSS animations provide visual feedback without JavaScript overhead:
  - animate-scan-line: A horizontal line scanning down the camera feed
    simulating active biometric scanning (communicates system activity)
  - animate-pulse: Applied to active indicators (camera on, processing)
  - animate-spin: Loading spinners for async operations
  - slide-in-from-bottom: Entry animations for status notification cards
  - animate-bounce: Attention-grabbing bounce on attendance confirmation notifications
  - animate-ping: Ripple effect on status indicator dots in calendar/log views

RESPONSIVE DESIGN:
  The layout uses CSS Grid and Flexbox through Tailwind's responsive prefixes:
  - Mobile (<768px): Single column, stacked layout
  - Tablet (768-1024px): Two-column main layouts
  - Desktop (>1024px): Three-column with sidebar navigation
  The primary use case is desktop/tablet as this is an institutional kiosk system.

ACCESSIBILITY:
  - All interactive elements have visible focus states
  - Status messages use both color AND iconography (not color alone)
  - ARIA labels on icon-only buttons
  - Sufficient color contrast ratios (WCAG AA minimum)

5.3 STUDENT REGISTRATION PORTAL (StudentRegistration.tsx)
----------------------------------------------------------

The registration portal is a multi-step wizard that guides administrators through
enrolling new students into the biometric database.

STEP 1: BASIC INFORMATION FORM
  Fields collected:
  - Student ID: Unique institutional identifier (format validation applied)
  - Full Name: Student's complete legal name
  - Email Address: Required for notification delivery
  - Department: Academic department (Computer Science, Electrical, etc.)
  - Mobile Number: Emergency contact
  - Date of Birth: For identity verification
  - Tutor Name: Academic mentor assignment

  Validation Rules:
  - Student ID must be unique in the database (checked via pre-existing records)
  - Email must match RFC 5322 format (HTML5 type="email" validation)
  - Name must contain at least 3 characters
  - Department must be provided for meaningful attendance department filtering

STEP 2: BIOMETRIC FACE CAPTURE
  The biometric capture step is the most technically complex part of registration.
  
  Camera Initialization:
  navigator.mediaDevices.getUserMedia({ video: { facingMode: 'user' } })
  returns a MediaStream that is displayed in a <video> element.
  
  Capture Process (3 frames required):
  - User positions face within the circular guide overlay
  - System runs detectSingleFace() on the live video frame
  - If face detected with sufficient quality: snapshot extracted to Canvas
  - Snapshot displayed in capture gallery row below video
  - Process repeated for 3 captures with slight head movements for variety
  
  Descriptor Averaging:
  Three 128-dimensional descriptor vectors are extracted from the three captures.
  The final stored descriptor is the mathematical average of the three vectors,
  providing a more robust representation than any single capture.
  
  Pseudo-code:
    captures = []
    FOR each capture attempt:
      frame = drawVideoFrameToCanvas()
      detection = await detectSingleFace(frame).withFaceLandmarks().withFaceDescriptor()
      IF detection found:
        captures.append(detection.descriptor)
        snapshots.append(canvas.toDataURL())
    
    IF len(captures) >= 3:
      finalDescriptor = average(captures)  // element-wise mean of Float32Arrays
      photoUrl = snapshots[0]  // first capture as profile photo
      PROCEED to step 3

STEP 3: CONFIRMATION AND STORAGE
  - Summary card displayed with all entered information
  - Photo preview of the student
  - "Complete Registration" button
  - On confirmation:
      registerStudent() called with all form data + descriptor + photoUrl
      Record inserted into students table in Supabase
      Registration confirmation email dispatched via Nodemailer
      Success message displayed, form reset after 2 seconds

DUPLICATE FACE DETECTION:
  Before allowing registration, the system checks for twin/similar face conflicts
  by comparing the new student's descriptor against all existing registered
  students. If any existing student has a distance < 0.40 (very high similarity),
  a warning is displayed to the administrator indicating potential duplicate
  or twin student scenario requiring manual verification.

5.4 STUDENT PORTAL (StudentPortal.tsx)
----------------------------------------

The Student Portal is the self-service attendance interface for students who are
already logged in with their credentials. Unlike the terminal (which is public-facing),
the portal knows the student's identity from the login session.

PORTAL FLOW:
  1. Student logs in with their Student ID
  2. Portal displays their identity card (photo, name, department, ID)
  3. After 3 seconds, automatically transitions to camera scanning step
  4. Camera activates and face detection begins
  5. Student looks at camera; system detects and matches their face
  6. On successful match (distance < 0.50):
     - First visit today: Records check-in, shows "ATTENDANCE IN VERIFIED"
     - Already checked in: Shows voice reason step for check-out
     - Already checked out: Shows "ALREADY ATTENDANCE TAKEN"
  7. On failed match (distance >= 0.50 for 3 consecutive attempts):
     - Shows "FACE NOT IN DATABASE (NO DATA FOUND)"
     - Sends security alert email to registered address
     - Automatically logs out after 3 seconds

THREE-RETRY SECURITY MECHANISM:
  failCountRef tracks consecutive failed verifications.
  On first and second failure: Shows retry message, resets after 1.5 seconds.
  On third failure: Triggers security alert, sends email, forces logout.
  This prevents unauthorized persons from attempting to game the system.

5.5 ATTENDANCE TERMINAL (AttendanceTerminal.tsx)
-------------------------------------------------

The terminal is designed for shared-workstation/kiosk deployment where students
approach a public attendance station and enter their ID manually (since no
pre-authentication has occurred).

TERMINAL FLOW:
  1. Student enters their Student ID at the ID entry screen
  2. System validates ID against database via loginUser()
  3. If valid: displays student identity card (same as portal step 2)
  4. After 3 seconds: camera activates
  5. System attempts face verification against student's stored descriptor
  6. On match: Same three-state attendance logic as portal
  7. On failure: "Face not found" error displayed for 10 seconds, then resets

KIOSK-APPROPRIATE DESIGN:
  - Large, touch-friendly buttons
  - Minimal navigation (only the attendance flow, no other system access)
  - Auto-reset behavior: returns to ID entry after every attendance event
  - Clear, high-contrast status messages readable from arm's length

5.6 ATTENDANCE TRACKING - LIVE ADMIN CAMERA (AttendanceTracking.tsx)
----------------------------------------------------------------------

This component is exclusively for administrative use. It provides a persistent
camera feed that automatically detects and records attendance for all students
who enter the camera's field of view simultaneously.

MULTI-FACE DETECTION:
  Unlike the student portal (single-face), the tracking view uses detectAllFaces()
  to process all faces present in the frame simultaneously. Each detected face
  is matched against the full student database. This allows a single camera
  at a classroom or corridor entrance to capture attendance for groups.

CANVAS OVERLAY:
  A transparent <canvas> element overlaid on the <video> element displays
  real-time bounding boxes around detected faces with color-coded status:
  - Green box + "VERIFIED" label: Successfully matched and attendance recorded
  - Amber box + "RECORDED" label: Student already has attendance today
  - Red box + "UNKNOWN" label: Face detected but not in database

  The drawOnCanvas() function renders these overlays using Canvas 2D API:
    ctx.strokeStyle = colorByType
    ctx.strokeRect(x, y, width, height)
    ctx.fillStyle = colorByType
    ctx.fillText(studentName, x, y)

TODAY'S LOG SIDEBAR:
